# CLAUDE.md — BakeMeter 開発ガイド

AIセッション(Claude Code等)がこのリポジトリで作業するときの前提知識。

## このリポジトリは何か

Thunderbolt eGPU 上で LLM を動かす Linux ホストの「凍結」を早期警報するシステムの**公開版**。
実測で得た知見(PCIe BadDLLP エラーが凍結の前兆になる)と、その対策スクリプトを共有するのが目的。

- 開発の本流は私有リポジトリ(local-ai-stack)側。ここには**較正済みの知見と汎用化したコードだけ**を反映する。
- 公開時は私有パス・ホスト固有情報を必要最小限に落とすこと(スクリプト内の `~` パスや PCI アドレスは「うちの例」として明示されている)。
- 実行環境はここには無い。本番は Mac Pro 2013 (TB2) + RTX 3090 上の cron で動いている。クラウドセッションでは動作検証できない。

## 構成

| パス | 役割 |
|---|---|
| `scripts/bake_meter.sh` | 本体 v2。cron で5分毎に実行。sysfs AER カウンタ(BadDLLP)の区間差分で水位判定し、危険遷移時に一度だけ全モデル退避+退避検証。`bake_state.json` をコンテナ可視の場所へ書く |
| `scripts/burst_test.sh` | 再現試験。LLM 風のバースト PCIe トラフィックを起こしてエラー率を測る |
| `docs/findings.md` | 実測知見 #1〜#9(番号付き)。新知見は番号を増やして追記 |
| `README.md` / `README.ja.md` | **対訳**。片方を変えたら必ずもう片方も更新 |

## 変更時に守ること

1. **README は両言語同時更新**(内容を一致させる)。
2. **閾値の一致**: `bake_meter.sh` の `WARN_5M=1000` / `DANGER_5M=5000`(sysfs)と journalctl フォールバックの 50/h・200/h は README 本文にも書かれている。どちらかを変えたら両方直す。閾値は較正中の暫定値。
3. **findings の流儀**: 実測データ(数値・日付)付きで書く。推測は推測と明記。既存番号は書き換えず追記。
4. コミットメッセージは日本語・「何をなぜ」を1行目に(既存ログ参照)。

## 設計上の重要事実(壊さないこと)

- **journalctl は使わない(計数用途)**: カーネルのレート制限で実測 ~34倍過少(findings #8)。計測は sysfs `aer_dev_correctable` の BadDLLP 行の**区間差分**が正。journalctl は読めない環境のフォールバック+参考記録のみ。
- **カウンタ後退 = 再起動**: sysfs カウンタはリブートでリセットされる。前回値より小さい場合は「起動後累積」を今区間の値として扱う(初回実行はベースライン記録のみで警報しない)。
- **退避は「危険への遷移時に一度だけ」**: 5分毎の連発退避は過去の不具合(監査#3)。退避後は `api/ps` で結果を再確認し、正直に記録する。
- **`bake_state.json` は下流の契約**: open-webui コンテナから読まれ、危険時のアプリ側ゲートに使われる。キー(`level`/`delta_5m`/`baddllp_total`/`journal_1h`/`source`/`ts`/`epoch`)の変更は互換性を壊す。原子的書き換え(tmp→mv)を維持。

## 危険な操作(クラウドセッションは実行禁止・実機でも注意)

- `burst_test.sh` は実機の TB リンクに意図的にストレスをかける(凍結リスクあり。ヘッダの監視手順を先に走らせる)。
- findings #7 の `echo c | sudo tee /proc/sysrq-trigger` は**マシンを故意にクラッシュさせる**訓練コマンド。

## テスト/CI

なし。シェルスクリプトは `bash -n` と shellcheck 相当の目視確認まで。実挙動の検証は実機(sysfs パスはホスト固有: 例 `0000:18:01.0`)でしかできない。
